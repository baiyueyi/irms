# Trae IRMS 前端代码规范 Rules

1. 页面分层必须清晰：page 负责页面编排与交互，components 负责可复用展示，services 只负责 API 请求与数据适配；禁止 page 直接写后端请求细节。
2. 所有接口调用统一走 `frontend/src/services/**`；禁止在 Vue 页面或组件内直接 `fetch/axios`。
3. 前端类型与结构必须分离：接口 request/response、页面展示对象、表单草稿对象分开定义；不要把同一个对象同时当 API DTO、表格行、编辑表单使用。
4. 编辑弹窗必须使用独立 draft state；禁止直接修改列表 row；取消即丢弃 draft，保存成功后才刷新列表或替换行数据。
5. 新增与编辑流程尽量复用同一套表单结构，但必须区分 createDraft / editDraft 初始化逻辑。
6. 环境标签、成员、多选项等复杂字段必须以草稿态为唯一数据源；禁止“已保存态 + 草稿态”双数据源混用。
7. 页面列表展示对象应尽量对应后端 VO；若需前端派生字段，使用明确的 mapper/adapter/computed，不要把后端原始对象直接改成展示对象。
8. 统一使用 mapper/adapter 做 DTO -> ViewModel 转换；禁止在模板里堆大量字段拼接逻辑。
9. 能直接复用后端返回字段的，不要在前端重复造一套业务语义；前后端冲突时优先推动后端 VO 收口。
10. 路由、菜单、页面资源路径必须保持一致；统一使用 canonical 路径，禁止页面里硬编码旧路径或分叉路径。
11. 列表查询参数、分页、排序、过滤要统一封装；不要每个页面各写一套 query 拼装逻辑。
12. 表格页优先复用通用 CRUD 能力；但如果通用层已经导致状态混乱、插槽过深、难维护，可以收敛而不是强行复用。
13. 通用组件只保留“真正通用”的能力；不要为了复用而把简单页面强行塞进过度抽象的组件体系。
14. `components` 不承载业务权限判断；权限判断放在 page 或 composable/service 层。
15. 复杂业务状态优先抽到 composables；但 composable 必须职责单一，不要做成隐藏全局状态仓库。
16. 不要滥用 watch；能用 computed、显式事件流、一次性初始化解决的，不要靠 watch 同步状态。
17. watch 只能处理明确副作用；禁止用 watch 在多个状态源之间来回回写，制造隐式联动。
18. 弹窗、抽屉、步骤表单等交互必须考虑响应式：桌面端可两列，小屏自动一列；禁止固定宽度把表单压坏。
19. 列表中的 tag、environment、member 等密集字段必须可读：允许换行、折叠、tooltip，但不要把整行挤成不可读长串。
20. API 错误处理要统一，页面不要各自手搓错误提示；成功/失败提示、401 处理、字段校验提示统一收口。
21. 所有页面跳转、详情跳转、权限管理跳转必须通过 router 或统一导航方法；禁止散落硬编码 URL。
22. 前端代码优先保证“状态流简单可解释”，不要为了形式优雅增加无价值抽象。
